5 visual improvements for HEY

October 12, 2020 | 4 min read

I tried HEY for the first time and these were a few things that jumped out right away. They, I believe, could have made my onboarding smoother and could have helped me to grasp the interface quicker.

I realize that you probably might have already thought of about some of these (if not all!) and they might be on their way. I understand that it's hard to make everything perfect from the start, especially when the top priority is to simply make things work properly.

Nevertheless, I wanted to share with you my observations. Visuals is also one my strongest qualities, so I didn't want to miss a chance to show it off :)

***

Styles for hover overs

The first thing I noticed was lack of style change when hovering over clickable elements, aside from the pointer cursor. While pointer cursor is undoubtedly a core indicator of clickable elements (although some might disagree...), it's always nice to get an extra feedback.

I saw a lot of various buttons on HEY without hover over states, which made me wonder whether it was a conscious choice. And if so, I would be very interested to learn the reasoning behind it.

Current implementation without hover over change

Suggestion - slight hover over background change

Visual separation between components

In day mode, white background of the page content sometimes blends with the white header. Because of that, it's easier to become disoriented when you're just learning the interface and still don't have a mental image of the app architecture.

However, this can easily be solved with a tiny gray line under the header, which will right away establish the border between the components within the fixed header and the page content itself.

Without the gray line

With the gray line

Current menu item indicator

Another thing that could be helpful for creating a mental map are distinct styles for the current menu item.

Example 1. Currently center nav menu doesn't have an indicator for the current menu item

Example 1. Suggestion: different background-color and font color for the current menu item

Example 2. Currently, when opening up the right nav menu, the highlight defaults to the first option.

Example 2. Suggestion: active menu option can have a different font color and/or a different shade of background highlight.

Border-radius for all

Rounding corners of elements makes the interface feel softer and more welcoming. When one decides to pursue this strategy though, it's important to make sure that it gets applied even to the most subtle elements. For example, while hover background for email list items is very subtle and transparent, it still should have border-radius in order to keep in sync with other components.

Currently there's no background border radius when hovering over email list item

Suggestion - 4px background border radius

Text vs Default cursor

This one is super minor and might be very specific to me personally, as I haven't noticed many other website and apps that do this. In short, I believe that "text" cursors should only be used in cases where you may want to type (input fields for forms, search) or, maybe, to copy text (email, article). In most other cases, the cursor should be "default" (headers, descriptions, tags). Given that "text" cursor is applied automatically to text, it always takes extra to change it to the "default" arrow cursor.

Despite that, I believe it's worth the effort since it helps one to stay focused and interact only with the parts that are necessary.

Currently, "text" cursor applies automatically when hovering over text blocks

Suggestion: "text" cursor is changed to "default" when hovering over text blocks
