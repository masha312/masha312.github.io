And this did work well, for some time.

However, as we started going farther down the roadmap and implementing more complex areas/features, things started to get messy. Unintentionally, we developed a new way of working which didn‚Äôt allow us to maintain the existing set up.

Here‚Äôs what happened:

  1. Every time we would start working on a new project, we would create a new file inside Component Library for exploration (in order to avoid cluttering up the Components file)
  2. We would create new components and prototypes in that new file
  3. We would then iterate in that file until we solidified the new design
  4. By the time we would be done with designs, it would be time to make specs for devs
  5. And after all that, it would be too late and inconvenient to transfer new components to the original Components file

As a result we ended up having numerous component files for different areas and features. Those files neither had a matching page structure nor a matching naming convention for the components.

Component Library v2

We had two problems to solve: first, organize the files and second, create a naming convention for components.

Problem 1: File organization

At first we considered transferring all existing feature-/area-specific components to our original Components file. However, that quickly proved to be a terrible idea for the following reasons:

  • It would be a huge time commitment
  • We would lose connections to all existing instances of those components
  • It might do more harm than good ‚Äî maintaining components will most likely become a pain + there‚Äôs a high chance that a large number of components will slow down the file
  • It‚Äôs not scalable. If we were to implement new features, we would most likely do what we found worked best before ‚Äî create a new file (and then the cycle would repeat‚Ä¶)

Therefore, we decided to keep the feature-/area-specific component files with the following adjustments:

  • Every file would have the same page structure
  • Every file would only have feature-specific or area-specific components. If a component is used across multiple areas, then it should be moved to the original Components file.

This ended up being a much, much better solution. While we still lost some instance connections, we ended up with cleaner (and smaller) files.

As for organization of pages within files, we decided to implement atomic design structure there as well ‚Äî with some flexibility.

Depending on the feature or the area, we created pages based on what made the most sense for that file. For example, a file for User Management only has ‚ÄúOrganisms‚Äù and ‚ÄúTemplates‚Äù pages, whereas a file for Workflows has ‚ÄúAtoms‚Äù, ‚ÄúMolecules‚Äù, ‚ÄúOrganisms‚Äù, and ‚ÄúTemplates‚Äù.

Pages for ‚ÄúUser Management‚Äù vs. ‚ÄúWorkflows‚Äù. Even though files may require different layers, they follow the same atomic structure.

We also ended up modifying our original Components (renamed to Universal Components) file:

  1. It now only includes ‚ÄúAtoms‚Äù, ‚ÄúMolecules‚Äù, and ‚ÄúOrganisms‚Äù. Any existing templates or pages were moved to new files related to that specific area or feature.
  2. It now also includes Colors, Typography, and Spacing. To be quite honest, there was no specific reason for moving it. Maybe it wasn‚Äôt a good idea..

Lastly, as you may have noticed we no longer include prototypes in our component files. We learned the hard way that it‚Äôs better to keep prototypes in a different file to make sure nothing breaks in case you modify components without intending them to affect the prototype.

Problem 2: Naming convention for components

I don‚Äôt think there‚Äôs a wrong way to do this, as long as you settle on one system that‚Äôs easy to remember when naming new components and makes it easy to search for existing ones. Here‚Äôs what we did:

  • We try to utilize Figma‚Äôs instance swapping feature whenever possible, by using the slash (/) structure. It helps to ensure that only related components can be swapped.

Conclusion

To summarize, here‚Äôs what we did:

  1. Created 3 color-coded categories of files in our component library: one for universal components (ideally only has one file), for feature-specific components, and for area-specific components.
  2. Standardized page structure across all files using atomic design methodology.
  3. Created a matching naming convention between files to utilize Figma‚Äôs instance swapping feature.
  4. The biggest difference between our previous component library set up and this one is that it allows for growth ‚Äî at least for the time being.
